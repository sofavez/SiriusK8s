# Лабораторная работа — namespaces, cgroups и chroot

## Блок 1. Работа с namespaces

Команда `ls -la /proc/$$/ns/` показывает, в каких namespaces находится текущий процесс.  
По выводу видно ссылки на `pid`, `net`, `mount` и другие пространства имён.

Команда `lsns` выводит список всех активных namespaces в системе и связанные с ними процессы.  
Это позволяет оценить, какие типы изоляции уже используются.  
![Скриншот 1 — вывод lsns со списком активных namespaces](1.png)

Команда  
```bash
sudo unshare --pid --fork --mount-proc /bin/bash
```  
запускает новый `bash` в отдельном PID‑namespace.  
Процессы внутри такого окружения не видят процессы хоста.

Команда `echo "Мой PID: $$"` внутри нового namespace показывает, что оболочка имеет PID 1.  
Такое поведение подтверждает, что создано изолированное пространство процессов.  
![Скриншот 2 — вывод PID 1 внутри нового PID namespace](2.png)

Команда  
```bash
sudo unshare --net /bin/bash
```  
запускает `bash` в новом сетевом namespace.  
Сетевые настройки такого окружения не связаны с настройками хоста.

Команда `ip link show` в новом сетевом namespace показывает только интерфейс `lo`.  
Отсутствие других интерфейсов подтверждает сетевую изоляцию.  
![Скриншот 3 — вывод ip link show с единственным интерфейсом lo](3.png)

## Блок 2. Ограничение ресурсов с помощью cgroups

Команда  
```bash
sudo mkdir /sys/fs/cgroup/mytest
```  
создаёт собственную cgroup, в которой можно задавать ограничения по ресурсам для процессов.

Команда  
```bash
echo "20000 100000" | sudo tee /sys/fs/cgroup/mytest/cpu.max
```  
устанавливает лимит на использование CPU.  
В таком режиме процесс может использовать примерно 20 % доступного процессорного времени.

Команда  
```bash
stress-ng --cpu 2 --timeout 60s &
```  
запускает тестовую нагрузку на два ядра в фоновом режиме для проверки действия лимита.

Команда  
```bash
echo $! | sudo tee /sys/fs/cgroup/mytest/cgroup.procs
```  
помещает запущенный процесс `stress-ng` в созданную cgroup.  
После этого процесс подчиняется установленным для группы ограничениям.

Команда  
```bash
cat /sys/fs/cgroup/mytest/cpu.stat
```  
отображает статистику работы лимита.  
Параметр `nr_throttled` показывает, что процесс регулярно замедлялся, что подтверждает применение ограничений.  
![Скриншот 4 — вывод cpu.stat с полем nr_throttled](4.png)

## Блок 3. Изоляция через chroot

Команда  
```bash
mkdir -p /tmp/myroot/{bin,lib,lib64,proc,dev}
```  
создаёт минимальную структуру каталогов для будущего изолированного окружения.

Команда  
```bash
cp /bin/bash /tmp/myroot/bin/
```  
копирует бинарный файл `bash` внутрь подготовленной директории.  
Эта оболочка используется как основная команда внутри `chroot`.

Команда  
```bash
ldd /bin/bash
```  
показывает список динамических библиотек, необходимых для запуска `bash`.

Команда  
```bash
cp /lib64/*.so* /tmp/myroot/lib64/
```  
копирует требуемые библиотеки в папку `lib64` нового окружения.  
Без этих файлов оболочка не сможет запуститься в изоляции.

Команда  
```bash
sudo chroot /tmp/myroot /bin/bash
```  
запускает `bash`, воспринимая каталог `/tmp/myroot` как новый корень файловой системы.  
![Скриншот 5 — запуск bash внутри chroot /tmp/myroot](5.png)

Команда `ls /` внутри chroot‑окружения показывает только те каталоги, которые были заранее созданы.  
Остальная часть файловой системы хоста в таком режиме недоступна и не отображается.

## Ответы на вопросы

### Почему после `exit` процессы хоста остались нетронутыми

Команда `exit` закрывает только временную оболочку или терминал, запущенный внутри namespace или chroot.  
Namespaces изолируют видимость процессов и ресурсов, но не создают отдельный физический компьютер.  
После выхода из окружения изолированные процессы либо завершаются, либо становятся недоступными из текущего терминала, тогда как все обычные процессы хоста продолжают работу в прежнем режиме.

### Что произойдёт при превышении лимита памяти

При превышении доступного лимита памяти для процесса в Linux срабатывает механизм OOM‑killer.  
Ядро анализирует потребление ресурсов и завершает процесс, который расходует слишком много памяти.  
Такой подход позволяет избежать зависания всей системы и сохранить работоспособность остальных процессов.
